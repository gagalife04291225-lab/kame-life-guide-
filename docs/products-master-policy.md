Phase 39-P1-Step3 で追加。対象:

- `research/products-master.sample.json`

## 研究用DBと公開DBの違い

| | 研究用DB（本ドキュメント対象） | 公開DB（`data/products.js`） |
|---|---|---|
| 目的 | 750商品候補の調査・スコアリング・昇格判断の材料 | サイトに実際に表示・アフィリエイトリンクを掲載する商品 |
| URLフィールド | **なし**（`amazon_status` / `rakuten_status` は状態のみ） | `affiliateUrl`, `asin`, `rakutenUrl` 等、実URLを保持 |
| id命名 | `RM0001`〜（研究用の独立ネームスペース） | `enclosure_wood_90` 等、カテゴリ＋特徴の命名規則 |
| 検証内容 | スキーマ整合性（必須項目・enum・スコア範囲） | 実商品ページ一致・在庫・価格・追跡ID |
| 更新頻度 | 調査が進むたびに随時 | 個別確認後にProject運営者の承認を得てから |

研究用DBはあくまで「候補リストと評価」であり、そのままサイトに表示されることはない。
`products.js` へ反映するには、下記の「昇格条件」をすべて満たす個別確認が必要。

## products.js へ昇格する条件

1. **実在商品ページの個別確認** — Amazon.co.jp または楽天市場の実際の商品詳細ページを開き、商品名・型番・ブランドが完全一致することを確認する
2. **在庫確認** — 確認時点で購入可能な状態であること（売り切れ・取扱終了は不可）
3. **価格帯の実測** — `price_band`（budget/standard/premium/unknown）ではなく、実際の価格帯（例: `¥3,000–6,000`）を記載する
4. **アフィリエイトURL生成** — Amazonは `tag=kamelife09-22` 付きURL、楽天は実商品ページの正規アフィリエイトリンクを発行する
5. **重複チェック** — 同一・類似商品が既に `PRODUCTS` に存在しないか確認する（本DBの `notes` に重複候補が記載されている場合は特に注意）
6. **CLAUDE.md のSKILLプロセスに従う** — 「HTMLページ制作」および該当ページの品質確認チェックリストを満たす

上記すべてを満たした商品のみ、`amazon_status` / `rakuten_status` を `available` に更新し、
`products.js` に実データとして追加してよい。

## 架空URL禁止

- 研究用DBには **いかなるURLフィールドも持たせない**。存在しない・未確認の商品ページURLを
  「とりあえず」で埋めることは禁止。
- `sources.md` に記載の一次情報源（メーカー公式サイト等）はブランド・製品仕様の裏付けに使うことはできるが、
  それをそのままアフィリエイトURLとして転記してはならない。

## 楽天検索URL禁止

- `https://search.rakuten.co.jp/search/mall/...` のような検索結果URLは、
  **公開DB（`products.js`）のアフィリエイトリンクとして使用してはならない**。
- 公開DBで検索フォールバックが必要な場合は、既存の `rakutenSearchTerm` + `getRakutenSearchUrl()`
  の仕組み（`data/products.js` 末尾のヘルパー関数）を使う。これは実行時に動的生成されるものであり、
  研究段階で確定URLとして書き込むものではない。

## 実在商品ページのみ許可

`products.js` に書き込んでよいURLは、以下のいずれかのみ:

- Amazon.co.jp の実際の商品詳細ページ（`/dp/{ASIN}?tag=kamelife09-22`）
- 楽天市場の実際の商品詳細ページを経由した正規アフィリエイトリンク

上記以外（メーカー公式サイトのトップページ、検索結果ページ、SNS投稿など）を
`affiliateUrl` / `rakutenUrl` に入れてはならない。

## 大量投入禁止

- 750件を一度に `products.js` へ投入することは行わない。
- 理由: `equipment-db-750.csv` は調査の網羅性を優先した候補リストであり、
  同一商品のサイズ・型番違いによる重複や、国内入手性が未確認の海外専用商品が多数含まれる
  （テンプレート的に生成された類似候補が連続する区間もある）。無検証のまま投入すると
  誤情報・在庫切れ商品へのリンク・追跡ID漏れのリスクが高い。

## まずは上位20〜50件ずつ昇格する運用

1. 本DBの `priority_score` / `quality_score` / `revenue_score` を基に上位候補を抽出する
2. `Phase38-A` の実績（`phase38-top20.csv` の上位20件からさらにTOP5を実装）にならい、
   一度に昇格させるのは **20〜50件を上限の目安** とする
3. 昇格作業は「昇格する条件」の6項目をすべて満たした商品から順に、少数ずつ `products.js` に追加する
4. 追加後は `git status` で差分を確認し、Scope Lock外のファイルを変更していないことを確認してから
   push する
