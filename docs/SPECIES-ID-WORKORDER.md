# 種識別データ作成 作業指示書（AI Company OS 分担用）

## 目的
`data/species-identification.json` を全104種ぶん埋める。このデータは**写真選別（qc_filter等）が混同種の区別点と写真判定基準を参照する**ためのもの。写真を選ぶ前に、まず識別特徴を確定させることが絶対の前提。

## 役割分担
- **各担当AI（ChatGPT / Copilot 等）**: 割り当てられた属・グループの種について、下記ルールに従い識別データの**草案**を作る。
- **検収・マージ担当（Claude）**: 全草案を照合・検収し、OKなら本番JSONにマージしてpushする。品質の最終責任はClaudeが持つ。
- **最終決定者（亀好きさん）**: 不可逆操作・方針判断。

---

## 絶対ルール（品質維持の核）

### R1. 複数ソースを必ず調べる（1ソース禁止）
最低でも次の3系統を確認し、食い違いを潰すこと:
1. **学術/百科**: Wikipedia（日英両方）、Animal Diversity Web、IUCN Red List、Reptile Database
2. **飼育/流通**: 信頼できるキーパー記事・専門ショップ（和名・流通名・"色型"の実態確認）
3. **一次情報**: 可能なら原記載・IUCN assessment・CITESチェックリスト

> 教訓例: マタマタをWikipediaだけで見ると "Black/Red variety" を別種と誤解する。ペットショップ記事まで見て初めて「業界の色型ラベルであって別種でない」と分かる。**1ソースは必ず罠がある**前提で照合する。

### R2. 混同種の区別は「相互一貫」で書く
種Aの `kondou_shu` に種Bを書いたら、種Bの `kondou_shu` にも種Aを書き、**区別点の記述が矛盾しないこと**。（例: サルヴィンとスジオオの「サイズ差」はどちらのページでも同じ数値で書く）

### R3. CITES・法規制は必ず一次情報で確認
「CITES掲載」と書く前に CITES Appendices / チェックリストで実際の掲載を確認する。
> 教訓例: マタマタ属・サルヴィン等を「CITES II」と誤記していた事故が実際にあった。**掲載していないのに掲載と書くのは重大な誤り**。掲載/非掲載/附属書番号を一次で確認。

### R4. 学名は三名法・亜種まで正確に
亜種ページは亜種名まで（例: Terrapene carolina triunguis）。**隣の種から学名をコピペして貼り替え忘れる事故が多発している**ので、slugと和名と学名が三者一致しているか必ず確認。

### R5. 写真判定基準は「写真に何が写っていれば同定できるか」を書く
`photo_hantei` には、その種を写真で正しく判定するために**どのカット・どの部位が必要か**を書く。
> 例: マタマタは「首下面の黒帯」が識別点なので、上からの甲写真だけでは同定不可 → 「首・喉の下面が見えるカット必須」と書く。

### R6. できたフリ・手抜きの禁止
- 「調べた」と言って調べていないのは禁止。確認したソースURLを `sources` に必ず残す。
- 一部の種だけ埋めて残りを見落とす手抜きは禁止。割り当て分は全種埋める。

### R7. 分からない種こそ徹底的に洗い出す（逃げ禁止）★最重要
**「難種だから曖昧」「情報が乏しいから空欄」は禁止。** 実際、過去の事故（hime-nioiの学名depressus誤り、giant-musk-mxのサルヴィン取り違え）は、すべて"難しいから曖昧に放置していた種"で起きた。**難種を避けると事故が温存される。**

分からない種は、決着するまで掘る:
1. **日英Wikipedia + Reptile Database + Animal Diversity Web** を横断
2. **原記載論文・分類学的改訂**（種が分割/統合された経緯。例: マタマタ2020分割、ヒメニオイ2017分割）を確認
3. **IUCN assessment・CITESチェックリスト** で保全/規制の一次情報
4. **標本写真・博物館記録・専門キーパーの繁殖レポート**（クリーパー誌等）まで当たる
5. 学名が疑わしい種は **同属の全種を並べて**、どれと取り違えやすいかを総当たりで確認（全104種学名監査と同じ手法）

それでも一次情報で確定できない項目がある場合のみ、**空欄で誤魔化さず**、`unresolved` に「何が・なぜ・どのソース間で食い違って不明なのか」を具体的に書く。**"分からない"を具体的に記録することも成果**。曖昧な空欄は成果ではない。

## JSONスキーマ（1種ぶん）
```json
{
  "slug": "リポジトリのファイル名",
  "wamei": "和名（流通名も併記可）",
  "gakumei": "学名（三名法）",
  "eimei": "英名",
  "koura": "背甲・腹甲の形態（キール数・甲形・鋸歯・ヒンジ等）",
  "iro_moyou": "色・斑紋（幼体と成体の差も。色型があれば明記）",
  "tobu": "頭部・頸部・四肢の特徴（指の数・髭状突起・斑紋等）",
  "size": "最大甲長・雌雄差",
  "bunpu": "分布（水系・州・国レベルで具体的に）",
  "kondou_shu": [
    {"aite": "混同種の和名(学名)", "kubetsu": "具体的な区別点。相互一貫で書く"}
  ],
  "houkisei": "CITES掲載状況（附属書番号 or 未掲載）・国内外規制。一次確認必須",
  "photo_hantei": "写真で同定するために必要なカット・部位",
  "photo_chuui": "写真選別で取り違えやすい注意点",
  "unresolved": "一次情報でも確定できなかった項目と食い違いの具体的内容（無ければ空文字）",
  "status": "confirmed（Claude検収済のみ） / draft（草案・未検収）",
  "sources": ["確認したソースURL（検収の証跡・最低3系統）"]
}
```

## 提出方法
- 担当AIは自分の担当分を**別ファイル**（例: `data/_id-draft-<担当名>.json`）に書き、同スキーマの配列で提出する。
- `status` は必ず `draft` で提出（`confirmed` はClaude検収後にのみ付与）。
- `sources` に確認URLを必ず入れる（最低3系統。R1/R6/R7の証跡）。

## 検収ゲート（Claude担当）
草案を受け取ったら、Claudeが以下を確認してからマージ:
1. R1: 複数ソース（最低3系統）が sources に入っているか
2. R3: CITES記述が一次情報と一致するか（実際に照合）
3. R4: slug/和名/学名の三者一致
4. R2: 混同種の相互一貫（相手ページと矛盾しないか）
5. R5: photo_hantei が「写る部位」を具体的に指定しているか
6. R7: `unresolved` が空でない場合、Claude自身が追加調査して決着を試みる（担当が掘り切れなかった所をClaudeが最終的に掘る）
- 全通過 → `status: confirmed` に変えて本番 `species-identification.json` にマージ・push
- 不備 → 差し戻し（どのRに反したか明示）
